Intelligence Operations Guide — MDO C4ISR Strategic Radar

        
    

    


        
This platform is an MDO C4ISR Strategic OSINT Radar designed to detect precursor indicators of kinetic operations by correlating signals across three domains: Cyber, Physical, and Information. All data is sourced exclusively from open-source intelligence (OSINT). Sensors run on independent background schedules, and the convergence score is recalculated every SCORE_REFRESH_SEC=60 seconds from the latest cached data. The SYNC button forces an immediate fetch and score update across all active sensors.


        
1. Multi-Domain Sensor Architecture (15 Sensors)


        
Cyber Domain (weight 0.50) — Highest-weighted domain. State-directed cyber operations typically precede kinetic action by 24–72 hours.

        
    
            
  • Cloudflare Radar API [poll: 15 min]: Covers over 20% of global internet traffic. Provides L3 (volumetric) and L7 (application-layer) DDoS telemetry. Calculates per-source-country spike ratios against a 7-day rolling baseline, detects botnet overlap across allied theaters, and identifies L7 vector shifts (HTTP flood transitions).
                
    Outliers capped at 25×. Sources with absolute share <1% excluded. Non-adversary floor: 3%; adversary floor (CN/RU/KP): 0.5% to retain low-volume state operations. Spike calculation skipped when 7-day baseline is insufficient (cold-start guard).
            
    • 
            
  • ThreatFox (abuse.ch) API [poll: 1 hr]: Queries IoCs (Indicators of Compromise) with APT tags published in the past 24 hours. Matches against APT tag or target country name. A hit indicates known APT infrastructure is active in association with the monitored theater. Requires THREATFOX_API_KEY (Auth-Key header).
    • 
            
  • RIPE Stat BGP Routing [poll: 30 min]: Monitors BGP prefix announcement count for the core theater via RIPE NCC. Triggers on prefix reduction >15% from the previous cycle — a known precursor to internet isolation operations (route withdrawal).
    • 
            
  • DDoS Acceleration Engine [no external poll — computed internally]: Calculates first-order (velocity) and second-order (acceleration) derivatives of the core theater threat score time series via linear regression over a DERIVATIVE_WINDOW=5-point rolling window. Declares an Ambush Pattern when acceleration Z-score (relative to its own rolling history) exceeds AMBUSH_ZSCORE_THRESHOLD=2.0 and both velocity and acceleration are positive — the mathematical signature of exponential escalation. Adds +2 to Cyber domain. Current rate of change is displayed in the VELOCITY meter.
    • 
            
  • GreyNoise Sensor [poll: 30 min] [v9 — Noise Suppressor]: Queries the GreyNoise GNQL Stats API to classify ongoing traffic toward the core theater as TARGETED, MIXED, or NOISE_DOMINANT (internet background scanners). When noise ratio exceeds 70%, all Cyber domain sensor scores are suppressed — preventing mass-scanner events from inflating threat levels during non-adversarial traffic surges. Operates passively (UNKNOWN) when no API key is configured. Requires GREYNOISE_API_KEY (Enterprise GNQL).
    • 
        


        
Physical/Logistics Domain (weight 0.30) — Covers infrastructure, airspace, maritime, and kinetic pre-strike indicators.

        
    
            
  • IODA API (CAIDA) [poll: 5 min]: Tracks BGP-based internet reachability and active probing (ping) success rates. Simultaneous degradation of BGP routing and probing success is classified as a critical infrastructure failure. Cross-validates with the BGP Routing sensor.
    • 
            
  • NASA EONET Wildfire API [poll: 1 hr]: Real-time wildfire and thermal event data from NASA's Earth Observing Natural Event Tracker. Flags events within ±10° of the theater center coordinates. Thermal anomalies near military infrastructure correlate with kinetic strikes, artillery fires, and pre-assault infrastructure destruction. No API key required.
    • 
            
  • OpenSky Network (Civil Airspace) [poll: 30 min]: Monitors civil aircraft count within airport bounding boxes for each theater. Builds a 20-cycle rolling baseline. Deviation ≥40% from baseline triggers ANOMALY (+2); <5% triggers CLOSURE (+3, near-total airspace shutdown). Score is automatically suppressed during severe weather based on OpenWeatherMap data.
                
    Anonymous API: 400 req/day. Set OPENSKY_CLIENT_ID/OPENSKY_CLIENT_SECRET (OAuth2 client credentials) to expand to 4,000 req/day.
    • 
            
  • PeeringDB / IXP Critical Nodes [poll: 4 hr]: Retrieves registration data for Internet Exchange Points (IXPs) associated with monitored theaters. Used as geographic context for interpreting BGP anomalies and physical infrastructure targeting. Displayed as yellow diamonds (◆) on the map.
    • 
            
  • ISR Hotspot Sensor (OpenSky Military ISR) [poll: 30 min]: Monitors 10 ISR_HOTSPOT zones within a 200 km radius — Taiwan Strait, East China Sea ADIZ, Spratly Islands, Korean Peninsula DMZ, Black Sea, Baltic Sea, and others. Aircraft are flagged as ISR patterns if they meet any of: (1) altitude >9,000 m and speed <160 m/s (high-altitude low-speed orbit), (2) squawk 7777 (government/military transponder), or (3) known ISR callsign prefixes (FORTE/JAKE/MYSTIC/RICO/TROLL, etc.). Detection of ≥3 aircraft simultaneously in the same hotspot triggers an ISR Surge, adding +2 to the Physical domain. Operates independently from the civil airspace sensor (targets straits and patrol corridors, not airports).
    • 
            
  • AIS Maritime Sensor (AISHub) [poll: 30 min]: Monitors vessels near 23 chokepoints — Taiwan Strait, Luzon Strait, Malacca Strait, Hormuz, Bab-el-Mandeb, Baltic Sea, Bosphorus, Suez, and others — using AISHub public position data. Detects two anomaly types: (1) AIS Dark Gap — a previously tracked vessel ceases AIS transmission for ≥AIS_DARK_GAP_THRESHOLD=3600 seconds (1 hour) near a chokepoint, consistent with intentional radio silence, submarine positioning, or covert survey vessels; (2) Stationary Anomaly — a non-commercial, non-fishing vessel (AIS type outside 60–89) drifting at <0.5 knots within 50 km of a chokepoint. Adds +1 to Physical domain. Dark Gap vessels appear on the map as red-halo anchor icons.
                
    A 2-second delay is inserted between each chokepoint request to comply with AISHub guest API rate limits.
    • 
            
  • Check-Host Survival Sensor [poll: 10 min] [v9]: Uses the check-host.net API to verify live reachability of critical national infrastructure URLs (government, parliament, central bank, energy grid) for each monitored theater from 5 geographically distributed global nodes (JP/US/DE/NL/FR). Two-phase API: POST check request → GET result (5 s delay). Status levels: OK (≥80% node success) / PARTIAL (30–79%) → +2 / BLACKOUT (<30%) → +3. High-latency responses above CHECKHOST_TIMEOUT_MS=3000 ms are penalized by 50%. Displayed as the SURVIVAL HUD indicator. Configure node list via CHECKHOST_NODES in config.env.
    • 
        


        
Information Domain (weight 0.20) — Propaganda and diplomatic tone analysis. The earliest leading indicator; typically shifts 48–96 hours before an operation.

        
    
            
  • GDELT TimelineTone API [poll: 30 min]: Retrieves media tone for the core theater from the Global Database of Events, Language, and Tone. Triggers an alert (+1) when the tone score falls below –15.0 relative to the 28-day rolling average. Reference values: Russia→Ukraine pre-conflict period was –12 to –15; day of invasion was –18 to –25. –15.0 is the threshold for active military-tension rhetoric. Auto-suppressed during severe weather.
    • 
            
  • RSS Narrative Burst Sensor [poll: 30 min]: Simultaneously scans 5 state-aligned RSS feeds — TASS (RU), Xinhua (CN), Mehr News (IR), RT, KCNA Watch (KP). Matches theater-specific tactical keywords (e.g., for Taiwan: "military measures", "ultimatum", "median line crossing", "live-fire drills") against article titles and descriptions. Z-score analysis is applied to normalized frequency (hit count ÷ total articles) rather than raw counts, using a 30-day rolling baseline. Alert thresholds: Z > NARRATIVE_ZSCORE_ALERT=2.0 → BURST (+1); Z > NARRATIVE_ZSCORE_CRITICAL=3.0 → CRITICAL_BURST (+2). Deduplication via normalized 60-character title key (O(N)). Baseline activates after ≥7 days of data accumulation (cold-start guard). Ignores steady-state propaganda; detects only statistically anomalous surges correlated with operational preparation cycles.
    • 
            
  • Telegram Mirror Sensor [poll: 15 min] [v9]: Scrapes public Telegram channel mirror pages (tgstat.com / telemetr.io fallback) for channels registered in THREAT_ACTOR_MAPPING (hacktivist groups mapped to theaters). No login or API token required. Extracts attack intent keywords (target, ddos, http flood, under attack, op, etc.) and target URLs (.gov / .mil / bank / energy domains). Detection levels: INTENT_DETECTED (+2) / TARGETS_FOUND (+1) / CLEAR. Detected channels and target URLs are displayed in the CHAIN panel. When intent is confirmed, a NARRATIVE_BURST event is registered in the escalation chain log. Configure keywords via TELEGRAM_ATTACK_KEYWORDS.
    • 
        


        
Noise Reduction & Human-in-the-Loop (HITL) Control

        
    
            
  • OpenWeatherMap (Weather Noise Filter) [poll: 30 min]: When a significant weather event (thunderstorm, blizzard, hurricane, etc.) is detected at the core theater, Physical domain alerts (IODA BGP failure, OpenSky airspace anomaly) and the Information domain alert (GDELT tone) are automatically marked SUPPRESSED and excluded from scoring. This prevents weather events from triggering false escalations.
    • 
            
  • GreyNoise (Automatic Noise Suppressor) [v9]: When GREYNOISE_API_KEY is configured and the GNQL Stats API classifies >70% of traffic toward the theater as internet background noise (scanners, crawlers, opportunistic probes), the suppress_confidence flag is raised. This automatically suppresses all Cyber domain sensor contributions, preventing scanner waves from being misread as targeted attack operations. Operates silently as UNKNOWN when no API key is set.
    • 
            
  • Analyst MUTE (HITL Override): Each sensor in the Evidence Panel has a [ MUTE ] button. Clicking it immediately excludes that sensor's score contribution and recalculates the convergence score. MUTE state is persisted to browser localStorage across sessions. Use this to suppress known false positives while preserving the underlying sensor record. When multiple sensors are muted, domain weights are correctly recalculated using only the remaining active sensors. All 15 sensors support MUTE.

3. Scoring Engine & Convergence Algorithm

        
The WeightedConvergenceEngine computes a raw threat score from all active, non-suppressed sensor signals, then adds domain convergence bonuses and a sequence bonus. The theoretical maximum score exceeds 22 points.


        
Raw Score Components

        
    
            
  • [Cyber]    cf_spike_core:       Core theater weighted-avg spike >2×(+1), >4×(+1), >6×(+1) — max +3
    • 
            
  • [Cyber]    cf_adversary_strike: Adversary state (CN/RU/KP) direct attack ≥4× spike AND absolute share >3% → +2
    • 
            
  • [Cyber]    cf_botnet_overlap:   Botnet overlap with allied theaters >30% → +1
    • 
            
  • [Cyber]    cf_vector_shift:     L7 avg spike >2.5× AND exceeds 1.5× L3 avg → +1
    • 
            
  • [Cyber]    cf_coordinated:      ≥2 strategic theaters simultaneously above 3× spike → +1
    • 
            
  • [Cyber]    ripe_bgp:            BGP prefix reduction >15% → +1
    • 
            
  • [Cyber]    threatfox:           APT IoC matching theater in past 24 hrs → +1
    • 
            
  • [Cyber]    ddos_acceleration:   Ambush pattern (2nd-derivative Z-score >2.0, positive acceleration) → +2
    • 
            
  • [Physical] ioda_bgp:            BGP/probing failure confirmed (after weather filter) → +1
    • 
            
  • [Physical] opensky:             Airspace ANOMALY (≥40% deviation from 20-cycle baseline) → +2
    • 
            
  • [Physical] opensky:             Airspace CLOSURE (<5% of baseline, near-total shutdown) → +3
    • 
            
  • [Physical] nasa_firms:          NASA EONET wildfire within ±10° of theater center → +3
    • 
            
  • [Physical] isr_hotspot:         ISR Surge ≥3 aircraft (strategic hotspot) → +2
    • 
            
  • [Physical] ais_maritime:        AIS Dark Gap or Stationary anomaly → +1
    • 
            
  • [Physical] check_host:         Infrastructure PARTIAL (30–79% node success) → +2 [v9]
    • 
            
  • [Physical] check_host:         Infrastructure BLACKOUT (<30% node success) → +3 [v9]
    • 
            
  • [Info]     gdelt:               GDELT media tone < -15.0 (28-day rolling avg) → +1
    • 
            
  • [Info]     rss_narrative:       Narrative Burst Z > 2.0 → +1; Critical Burst Z > 3.0 → +2
    • 
            
  • [Info]     telegram_mirror:    Target URLs found in hacktivist Telegram channels → +1 [v9]
    • 
            
  • [Info]     telegram_mirror:    Attack intent intercepted (keyword match) → +2 [v9]
    • 
            
  • [Info]     maskirovka_flag:    Physical outage confirmed but all narrative channels silent → +1 [v9]
    • 
        

        
■ = v9 additions. greynoise sensor contributes score 0 (suppressor only — when NOISE_DOMINANT, all Cyber entries are excluded from the raw score total).


        
Domain Convergence Bonus — Added when anomalies appear simultaneously across multiple domains (evidence of coordinated MDO activity rather than an isolated incident):

        
    
            
  • DUAL CONVERGENCE: Any 2 of 3 domains simultaneously active → +1 pt
    • 
            
  • FULL CONVERGENCE: Cyber + Physical + Info all active simultaneously → +2 pt
    • 
        

        
A domain is considered "active" when its weighted score contribution exceeds the domain threshold. FULL CONVERGENCE is the strongest pre-kinetic signal this system can generate.


        
Sequence Bonus — Applied on top of the convergence bonus when a logical escalation chain is confirmed within a 24-hour window. Acts as an anti-confirmation-bias mechanism (does not fire on dramatic signals in a single domain):

        
    
            
  • Chain definition (loose ordering — presence within 24 hr window, strict chronological order not required):
                
      [1] NARRATIVE_BURST  — Tactical keyword Z-score spike in state media or Telegram intent detection
                
      [2] ISR_SURGE        — Reconnaissance aircraft detected at strategic hotspot
                
      [3] SYNC_DDOS        — Coordinated DDoS from ≥2 theaters (high botnet overlap)
                
      [4] FIRMS_ANOMALY    — Thermal/kinetic event from NASA EONET
    • 
            
  • 4/4 confirmed (within 24 hr) → SEQUENCE_FULL_BONUS = +3 pt
    • 
            
  • 3/4 confirmed (within 24 hr) → SEQUENCE_PARTIAL_BONUS = +1 pt
    • 
            
  • <3 events → No bonus (insufficient pattern)
    • 
        


        
Temporal Coherence Bonus [v9] — Applied when first trigger events (SYNC_DDOS or NARRATIVE_BURST) across multiple theaters are temporally synchronized, indicating a single C2 node:

        
    
            
  • C2_SYNC_CONFIRMED  — ≥2 theaters trigger within 60 seconds of each other → +2 pt
    • 
            
  • C2_SYNC_PARTIAL    — ≥2 theaters trigger within 5 minutes of each other → +1 pt
    • 
            
  • Displayed in the HUD as C2-SYNC. Distinguishes nation-state coordinated operations (tight synchronization) from independent hacktivist campaigns (loose timing).
    • 
        


        
Blockade Index (HUD display) — A gauge to quantitatively distinguish "political signaling" DDoS from actual infrastructure neutralization:

        
    
            
  • Formula (v9): Blockade Index = min(DDoS_spike, 10) × (1 + RIPE_drop_pct/100) ÷ max(CheckHost_success_rate, 0.05)
    • 
            
  • Previous formula used a fixed IODA flag divisor; v9 replaces it with live RIPE BGP drop ratio and Check-Host success rate for continuous granularity. RIPE at 0% drop → no amplification; at 100% → 2× amplification. Check-Host at 0% success → maximum denominator pressure (÷0.05 cap).
    • 
            
  • NORMAL                        — Index < 1.5: DDoS present but infrastructure reachable. Typical signaling operation.
    • 
            
  • POLITICAL_NOISE               — 1.5–3.9: Significant cyber activity, infrastructure impact unconfirmed.
    • 
            
  • SEVERE_DISRUPTION             — 4.0–6.9: High DDoS + RIPE prefix reduction + partial Check-Host failure simultaneously.
    • 
            
  • INFRASTRUCTURE_NEUTRALIZATION — ≥ 7.0: Large-scale DDoS + BGP withdrawal + infrastructure blackout. Signature of a pre-invasion communications cutoff.
    • 
        


        
4. Threat Levels

        
Final Score = Raw Score + Convergence Bonus + Sequence Bonus. A 1-cycle hysteresis delay is applied on downgrade only (prevents threshold oscillation). Upgrades are immediate.

        
    
            
  • Level 1 — CRITICAL: Score ≥ 9 AND hard conditions met (core spike >5.0× AND BGP failure confirmed simultaneously). Combined signature of large-scale cyber surge and infrastructure collapse — coordinated pre-invasion blackout operation. Kinetic action probability is extremely high. Activate all emergency procedures.
    • 
            
  • Level 2 — SEVERE: Score ≥ 6. Multi-domain threat with direct adversary attribution or significant infrastructure degradation. Immediate commander notification required.
    • 
            
  • Level 3 — HIGH: Score ≥ 4. Significant coordinated anomalies spanning at least 2 sensors. Indicates preparation phase, probing operations, or initial escalation. Enhanced monitoring required.
    • 
            
  • Level 4 — ELEVATED: Score ≥ 2. Early indicators such as deteriorating media tone, isolated BGP events, or minor L7 vector shifts. Consistent with worsening diplomatic tensions or reconnaissance activity.
    • 
            
  • Level 5 — NORMAL: Score < 2. Background noise level. No significant anomalies across monitored domains.
    • 
        


        
5. Threshold Calibration Reference

        
System thresholds are calibrated against the following confirmed historical cases:

        
    
            
  • Russia → Ukraine (24 Feb 2022): BGP routes reduced 10–15% in the 6 hours before invasion. HermeticWiper malware deployed 90 minutes before ground forces crossed. GDELT during pre-conflict period: –12 to –15; day of invasion: –18 to –25. Civil flights ceased 24 hours prior.
    • 
            
  • China → Taiwan Strait (Aug 2022, Pelosi visit): CN-origin DDoS against TW government and media infrastructure spiked 3–5×. GDELT tone during exercises: –13 to –17. No BGP failure (deliberate — PRC signaling, not isolation). Botnet overlap with JP/US-targeted traffic confirmed in post-analysis.
    • 
            
  • DPRK Persistent Operations (ongoing): KP-origin traffic maintains an absolute share of 0.1–0.5%. The adversary 0.5% baseline floor is set specifically to retain these low-volume, high-significance operations that would be undetectable at the 3% non-adversary floor.
    • 
            
  • Iran Internet Shutdown (2019): Near-complete BGP route withdrawal before domestic elections — a full isolation pattern. Corresponds to INFRASTRUCTURE_NEUTRALIZATION on the Blockade Index.
    • 
            
  • Georgia-Russia (Aug 2008): 5-day cyber campaign preceding the ground invasion. GDELT tone dropped from –10 to –14 in the lead-up, then fell below –18 coinciding with the ground assault.
    • 
        


        
6. Analyst Workflow & Best Practices

        
    
            
  • Prioritize cross-domain corroboration. Do not escalate on single-domain indicators alone. Civilian botnets regularly produce 3–5× DDoS spikes without any military significance. The minimum standard for recommending a Level 2 escalation is FULL CONVERGENCE (Cyber + Physical + Info).
    • 
            
  • Suppress known false positives via HITL MUTE. When a non-adversarial source is confirmed (e.g., airspace anomaly from a major sporting event, BGP shift from legitimate CDN maintenance), mute that sensor in the Evidence Panel. MUTE state persists across reloads via localStorage.
    • 
            
  • Use SITREP for commander reporting. The SITREP button generates a structured situation report including domain breakdown, sensor evidence, and trend indicators.
    • 
            
  • Verify baseline sufficiency. Spike ratios for Cloudflare activate after the 7-day rolling baseline is satisfied (conservative suppression on startup). RSS Narrative similarly requires ≥7 days of data. Check per-sensor baseline status in the Data Status panel.
    • 
            
  • Read the VELOCITY / AMBUSH meters. VELOCITY is the current rate of change of the threat score (pts per polling cycle ≈ 900 s). Positive (red ↑) = actively deteriorating. Negative (green ↓) = improving. ⚡ AMBUSH (red blink) = the rate of change itself is accelerating — a second-derivative spike. This is the most time-critical signal: it indicates a rapidly compressing attack window.
    • 
            
  • Read the escalation narrative in the CHAIN panel. The CHAIN button (HUD top-right) opens the evidence chain timeline. Displays timestamped [NARRATIVE_BURST → ISR_SURGE → SYNC_DDOS → FIRMS_ANOMALY] escalation events within the past 24 hours. FULL CHAIN (+3 pt) means all 4 event types appeared within 24 hours — a strong indicator of coordinated multi-domain preparation. v9 also registers TELEGRAM_INTENT, MASKIROVKA, C2_SYNC, and INFRA_BLACKOUT in the chain log.
    • 
            
  • Interpreting GDELT tone. –15.0 = threshold for active military-tension rhetoric (v9 calibration). –18 = active crisis framing. Below –20 = historically corresponds to kinetic action imminent or in progress. A 3–5 point drop within a single 30-minute polling cycle is more significant than a gradual decline to the same value.
    • 
            
  • Interpreting the Blockade Index (v9). High-volume DDoS that does not degrade BGP reachability or Check-Host availability is a signaling operation (Index < 1.5). INFRASTRUCTURE_NEUTRALIZATION (Index ≥ 7.0) requires simultaneous DDoS surge, BGP prefix withdrawal, AND live infrastructure confirmation from global nodes — the pattern observed 30–90 minutes before the Russian ground invasion in February 2022.
    • 
            
  • Interpreting the SURVIVAL HUD (v9). Shows real-time reachability of critical national infrastructure (government, central bank, energy grid) from global nodes. OK = infrastructure reachable. PARTIAL = some nodes report failure — possible DDoS effect or localized disruption. BLACKOUT = critical infrastructure unreachable from multiple global vantage points simultaneously. BLACKOUT + high Blockade Index = likely pre-invasion blackout scenario.
    • 
            
  • Interpreting C2-SYNC (v9). Displays whether attacks across multiple theaters started within 60 seconds (C2 SYNC CONFIRMED) or 5 minutes (PARTIAL). Tight temporal synchronization across geographically separated theaters is the hallmark of nation-state integrated C2 (command and control) rather than loosely coordinated hacktivist activity.
    • 
            
  • Maskirovka detection (v9). If physical outage is confirmed (IODA or Check-Host BLACKOUT/PARTIAL) while all narrative channels are simultaneously silent (no RSS burst, no Telegram intent), the system flags a potential Maskirovka (Russian military deception doctrine). Physical disruption without accompanying narrative amplification may indicate a genuine kinetic operation rather than a hacktivist campaign — hacktivist groups typically announce targets loudly before and during attacks.
    • 
        


        
7. Config & Monitoring Management

        
Open the master settings modal via the Config button in the toolbar. It consists of five tabs.

        
    
            
  • Strategy Scope: Epicenter (Core) is the primary focus theater (single selection). Nearly all sensors and the convergence score are computed relative to Core. Correlate (Link) selects allied nations monitored for coordinated attacks from the same botnet. The monitored universe covers ~127 countries retrieved via Cloudflare.
    • 
            
  • Threat Actors: Select adversary nations to monitor for direct strike attribution. When a designated nation's ASN is used for high-volume traffic, a state-attribution badge is displayed.
    • 
            
  • Quick Pins: Select countries to appear as quick-toggle buttons in the "Target Visibility" panel on the main screen.
    • 
            
  • Sensors: Enable or disable individual sensors. A disabled sensor contributes zero to the score (graceful degradation). v9 adds 3 sensors: telegram_mirror (Info), check_host (Physical), greynoise (Cyber — suppressor).
    • 
            
  • Fetch Log: Displays the last fetch result for each sensor (success/failure/cache age). Cache older than 3× the polling interval is flagged as STALE.
    • 
        

        
Region Filter (Region Pills): The Strategy Scope, Threat Actors, and Quick Pins tabs each include region filter pills (East Asia / SE Asia / Middle East, etc., across 16 regions). Clicking a pill zooms the Region Preview mini-map on the right to that region and displays currently selected countries as dot markers color-coded by role: Core (cyan) / Link (amber) / Adversary (red) / Pin (green). Pill selection persists across tab switches.


        
8. Intuition UI — Recognition-Primed Decision (RPD) Components

        
Eight visual components designed to engage analyst intuition and pattern-recognition (Recognition-Primed Decision) in parallel with numerical computation. Based on Naturalistic Decision Making (NDM) theory; intended for immediate situational awareness formation.


        

            
How to access

            
The TOOLS ▾ dropdown (HUD top-right) allows individual activation of 6 panels: Weather Brief / SALUTE Board / Historical Analog / Ops Clock / GreyNoise / Analyst Notebook (cyan = active). Close each panel individually with ✕. Always-on components (Pulse Display / Radio Silence / Ambient Atmosphere / Threat Terrain) operate automatically without any interaction. The Telegram SIGINT panel is opened via the SIGINT↗ button on the Telegram Mirror row in the CHAIN panel.

        


        
A. Ambient Atmosphere

        
    
            
  • Always active. Subtly shifts the background color of the entire screen in response to threat level (Lv3 → green tint / Lv2 → red-amber tint / Lv1 → deep crimson tint, 2–5 second animation cycle). Leverages peripheral vision to signal "something is wrong" during long monitoring sessions, preventing attentional tunneling.
    • 
        


        
B. Pulse Display

        
    
            
  • Always displayed at bottom-left of map. Renders threat level history (up to 60 cycles ≈ 15 hrs) as a scrolling EKG waveform (Canvas). Level 1 at waveform top, Level 5 at bottom. Waveform color tracks the current Threat Level color. Rate of change (/hr) displayed top-right.
    
            Reading: Rapid upward trend → situation deteriorating quickly. Flat plateau at high level → sustained high-threat environment. Downward turn → signs of de-escalation.
    • 
        


        
C. Operational Clock — TOOLS ▾ → Ops Clock

        
    
            
  • Displays Zulu (UTC) time in large real-time font (second-accurate). Simultaneously shows local time, UTC calendar date, and elapsed time since the last significant event (Level 3 or below). Provides a reference to distinguish genuine inactivity from tactical silence.
    • 
        


        
D. Operational Weather Brief — TOOLS ▾ → Weather Brief

        
    
            
  • Displays the threat state of each domain (Cyber / Physical / Info) using weather metaphors (e.g., Cyber = "Cyber Storm" = large-scale DDoS spike; Physical = "Strait Fog" = AIS dark gap active; Info = "Rising Pressure" = narrative operation intensifying). Enables intuitive understanding without domain expertise, increasing decision speed.
    • 
        


        
E. SALUTE Board — TOOLS ▾ → SALUTE Board

        
    
            
  • Auto-generates a military-standard SALUTE contact report from current threat state:
                
      
                    
    • Size       — Enemy force scale (e.g., 12 botnet sources, 3 state-ASNs)
      • 
                    
    • Activity   — Confirmed activity type (e.g., L7 coordinated surge + narrative burst)
      • 
                    
    • Location   — Target area (e.g., TW; correlated: JP/US)
      • 
                    
    • Unit       — Attribution estimate (e.g., CN attributed — state-ASN match)
      • 
                    
    • Time       — Time of first observed activity (UTC)
      • 
                    
    • Equipment  — Means employed (e.g., L3 volumetric + BGP withdrawal + ISR surge)
      • 
                    
    • +Assessment — Integrated assessment statement from current threat state
      • 
                
    
            
    • 
        


        
F. Historical Pattern Analog — TOOLS ▾ → Historical Analog

        
    
            
  • Compares the current threat score time series (latest 20 cycles) against 5 historical case patterns using Pearson correlation, and displays the top 3 matches:
                
      
                    
    • 🔴 Ukraine Invasion 2022 / 🟠 Taiwan Strait 2022 / 🟡 Iran Shutdown 2019 / 🔵 Georgia-Russia 2008 / ⬜ Normal Baseline
      • 
                
    
            
    • 
            
  • Each card shows similarity % and an overlay graph of the historical pattern (dim) vs. current pattern (cyan). Pearson ≥70% = statistically noteworthy pattern match. Correlation ≠ causation: this is a decision aid, not a prediction.
    • 
        


        
G. Radio Silence Indicator

        
    
            
  • Always displayed in the lower HUD. Detects a state where the threat score is significant (≥3 pts) but velocity is near zero — flagged as QUIET.
    
            QUIET implications: (1) pre-kinetic communications blackout (EMCON), (2) sensor suppression (adversary jamming/spoofing), (3) tactical operational pause. True silence does not mean peace — silence itself is treated as an anomaly to prevent the false perception that "nothing is happening."
    • 
        


        
H. Threat Terrain Overlay

        
    
            
  • Map layer control (top-right) → Threat Terrain. Renders semi-transparent choropleth-style circles over monitored countries scaled to DDoS spike intensity (avg_spike). BGP-confirmed failure = red; spike ≥4× = orange; lower = yellow. Radius proportional to spike intensity (max ~620 km). Allows threat pressure distribution to be perceived geographically at a glance. Countries at or below normal spike levels are not rendered, minimizing noise.
    • 
        


        
I. GreyNoise Investigator [v9] — TOOLS ▾ → GreyNoise

        
    
            
  • Theater Noise Classification: Shows per-theater GreyNoise classification — TARGETED (traffic is directed, not background scanners), MIXED, or NOISE_DOMINANT (≥70% internet background scan traffic, Cyber domain suppressed). Requires GREYNOISE_API_KEY (Enterprise GNQL). Without an Enterprise key, classification shows UNKNOWN.
    • 
            
  • IP Investigator (Community API): Manual on-demand lookup of any IPv4 address via the GreyNoise Community API. Returns: noise flag (true = internet-wide scanner), riot flag (known benign infrastructure), classification, entity name, and last-seen date. Results are cached 24 hours per IP. Rate limit: 50 req/day (Community plan). Requires GREYNOISE_API_KEY. Each lookup is automatically logged to the Analyst Notebook.
    • 
            
  • Investigation Log: Maintains a session history of the last 20 IP lookups, persisted to browser localStorage. Use this panel to triage suspicious source IPs surfaced in the Live Threat Telemetry panel.
    • 
        


        
J. Analyst Notebook [v9] — TOOLS ▾ → Analyst Notebook

        
    
            
  • Analyst Assessment: Dropdown (NOMINAL / WATCHFUL / CONCERNED / CRITICAL / HOSTILE) and 5-dot confidence scale (1 = LOW … 5 = HIGH). Records the analyst's current situational assessment independent of the automated threat score. Persisted across sessions.
    • 
            
  • Manual Notes: Free-text entry for recording observations, hypotheses, and reasoning. Press Ctrl+Enter or click LOG to append to the shift log. Each entry is timestamped (UTC).
    • 
            
  • Auto-logging: Three event types are logged automatically without analyst action:
                
      
                    
    • DEFCON — Any threat level change (e.g., "Threat Level ▲ 4 → 3 (score: 6)") is recorded immediately.
      • 
                    
    • IP_CHECK — Each GreyNoise IP lookup result (IP, noise flag, classification) is logged automatically.
      • 
                    
    • MUTE / UNMUTE — Sensor mute/unmute actions are logged with the analyst-provided reason (optional prompt dialog).
      • 
                
    
            
    • 
            
  • Watch Handoff: The "WATCH FOR" field at the bottom persists a handoff note for the next analyst shift. Use EXPORT to copy the full shift log to clipboard for shift turnover or incident reporting.
    • 
        


        
K. Telegram SIGINT — CHAIN panel → SIGINT↗ button on the Telegram Mirror row

        
    
            
  • Purpose: A dedicated floating/dockable panel that surfaces Telegram Mirror sensor intelligence in a structured SIGINT format — combining real-time status, theater activity breakdown, a scrollable intercept log, and the monitored channel roster.
    • 
            
  • Status Badge: Shows the current sensor state — INTENT DETECTED / TARGETS FOUND / CLEAR / PENDING — plus the last-poll timestamp and success/failure indicator.
    • 
            
  • Theater Activity Grid: One tile per active theater. Each tile displays the theater code, total hit count, and highest detection level badge. Empty-state text is context-aware: "Sensor not yet polled" before the first cycle; "No channel activity detected" once polling has completed with no hits. Click any tile to filter the intercept log to that theater only.
    • 
            
  • Intercept Log: A chronological ring buffer of up to 50 detections, persisted across polls in memory (cleared on server restart). Each entry shows channel name (linked), timestamp (UTC), detection type badge, matched keywords, targeted government/military domains, and a 200-character text snippet showing the surrounding context. Entries are color-coded: red border = INTENT, amber = TARGETS. Click a channel name to open the mirror URL. Use the [CLEAR] button in the log header to wipe the log (calls POST /api/telegram_log/clear). Filter by theater tile or [All Theaters] to reset.
    • 
            
  • Channel Roster: Lists all channels configured in THREAT_ACTOR_MAPPING with their associated theater and mirror URL. Use as a reference to verify channel coverage and add new channels to monitoring.
    • 
            
  • False-Positive Suppression: The sensor applies word-boundary regex matching (\b) to avoid substring collisions (e.g., "target" matching "targeting"). Keywords ≤ 2 characters are skipped entirely. Multi-word keywords and hashtags use exact substring matching.
    • 
        


        
9. API Reference (Analyst / Integration Endpoints)

        
    
            
  • GET /api/threat_data?core=TW&correlates=JP,US&adversaries=CN,RU — Primary data feed. Returns all sensor data, threat score, domain breakdown, and deep analytics block (including v9: temporal_coherence, maskirovka, check_host, telegram_mirror, greynoise).
    • 
            
  • GET /api/alert_timeline?limit=288 — Threat level history (up to 24 hrs / 288 cycles). Each cycle includes velocity, ambush flag, and blockade index.
    • 
            
  • GET /api/sitrep — Machine-readable SITREP including trend, domain activity, and convergence state.
    • 
            
  • GET /api/data_status — Sensor health, polling intervals, last fetch time, and error logs. Shows all 15 sensors including v9 additions.
    • 
            
  • GET /api/sensor_config — List of all sensors with enabled/domain/health status.
    • 
            
  • POST /api/sensor_config — Enable/disable a sensor. Body example: {"name":"telegram_mirror","enabled":false}
    • 
            
  • GET /api/sequence_chain?theater=TW — Returns escalation chain events within the 24-hour window for the specified theater. v9 event types: TELEGRAM_INTENT, MASKIROVKA, C2_SYNC, INFRA_BLACKOUT (in addition to NARRATIVE_BURST, ISR_SURGE, SYNC_DDOS, FIRMS_ANOMALY, AIS_DARK_GAP).
    • 
            
  • GET /api/deep_analytics?theater=TW — Deep analytics diagnostic dump: velocity series, acceleration, ambush Z-score, narrative Z-score, ISR aircraft count, AIS dark gaps, and Blockade Index (v9 formula) with interpretation label.
    • 
            
  • GET /api/ip_check?ip=1.2.3.4 — GreyNoise Community API IP lookup. Returns noise/riot/classification/name/last_seen. Results cached 24h per IP. Rate limit: 50 req/day (Community plan). Available via GreyNoise Investigator panel in TOOLS menu.
    • 
            
  • GET /api/salute_report — Generates a SALUTE-format contact report from current threat state. Used by the SALUTE Board panel.
    • 
            
  • GET /api/weather_brief — Maps sensor values to weather metaphors. Returns per-domain state, color, icon, and summary text. Used by the Weather Brief panel.
    • 
            
  • GET /api/historical_events — Returns the historical case pattern library from geo_data.json (5 reference fingerprints). Used by the Historical Analog panel.
    • 
            
  • GET /api/app_config — Returns default settings and the global monitored country list (with code, name, region, and coordinates). Used for UI initialization on startup.
    • 
            
  • POST /api/telegram_log/clear — Clears the in-memory Telegram SIGINT intercept log ring buffer. Used by the [CLEAR] button in the Telegram SIGINT panel.
